# 如何写好提示词

提示词工程，十分的基础，因为说白了是个人都会写提示词，但是写好提示词也还是有一些讲究的，不可不察。写好提示词的秘诀就是降低模型的不确定性，尽可能减少模型自由发挥的空间。

## 不要让模型猜

事情一定是有前因后果的，一定要尽可能地说明当前任务的背景、意图、目标。比如让`claude code`给你生成一份简历，那`claude code`肯定会问你额外的信息，因为你什么都不提供，没有人知道你想要什么，模型并没有读心术。但是如果你告诉`claude code`，

> “我目前正在应聘`java`后端开发岗位，简历上需要写`springboot`、`mybatis`相关的技术，需要放一个黑马点评项目到简历上。。，最终请给我一个`.html`文件样式的简历”。

这个例子虽不是很恰当，但已经足够，交代了背景，自己的需求和意图，以及最终想要得到的结果形式。

## 让模型抄答案

`few-shot`，也就是少样本策略，是一种简单但非常有效的方法，说白了就是给出几个示例，让模型进行模仿输出，而模型模仿输出的能力是非常强的。当你想让模型按照具体的格式输出时，不妨给出几个示例，让模型进行仿照学习。因此你给出的示例的质量一定要是比较好的，不要格式混乱，每个分隔符可能都会严重影响模型的判断，格式一定要一致。

## 模型的`cosplay`

给模型设定一个角色的力量是巨大的，就好比我们人类的自我暗示，每天暗示自己一定行，一定会成功的，那么就真的很有可能会成功。给模型一个角色的威力在于，可以迅速激活模型内在的相关知识。比如告诉模型：你是一个十年以上的大厂资深后端开发工程师。模型会迅速从自己训练的语料中激活相关知识。如果有一个更加具体一点的人物`cos`，是更好的，比如告诉模型：以林纳斯·托⽡兹审查内核补丁的⽅式来做代码审查。一般逼格越高，效果就越好。如果你告诉模型，你只是个根本不会`Python`的初学者，那么模型写`Python`代码的能力会即可崩塌。

## 结构化的威力

一种十分流行的做法是使用`XML`标签来给提示词划分边界。`XML`标签天生就带有语义，模型十分好理解，这是天然的边界。比如

`<role>你是一名资深的网络空间安全专家，擅长恶意代码的识别和逆向工程。</role>`

`<task>分析这个样本是否属于恶意代码。</task>`

边界和功能一目了然。
